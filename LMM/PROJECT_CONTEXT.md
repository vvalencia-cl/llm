## Goal of the app

Build a small **Windows** desktop app in **C# WinForms** that takes a **Word mail-merge template (.docx)** and an **Excel workbook (.xlsx)** and produces **one PDF per Excel row**. The PDF is generated by merging values into the template and exporting via **Microsoft Word Interop**. Output PDFs are saved to a user-selected folder and named as `FieldX_FieldY.pdf` where FieldX/FieldY are user-selected Excel columns.

---

## Architecture

- **UI:** Windows Forms (code-created layout; could be Designer-based later).
- **Pattern:** “layered utilities + orchestration”
    - UI layer (MainForm): gathers inputs, validates, launches run, shows progress/log, supports cancel.
    - Services/Helpers:
        - `ClosedXmlDataSourceService` (Excel sheet list, header extraction, formatted record enumeration with row number)
        - `WordTemplateFieldScanner` (extract merge field names from template)
        - `HeaderFieldMapper` (maps Word merge field names to Excel headers using robust normalization)
        - `OpenXmlMailMerge` (replaces MERGEFIELD results in DOCX main body)
        - `WordPdfExporter` (Interop wrapper: open DOCX, ExportAsFixedFormat PDF, cleanup)
        - `MailMergeToPdf` (pipeline: copy template → merge → export → cleanup)
        - `PdfFilenameBuilder` (safe Windows filename creation)
        - `StaTaskRunner` (runs the whole Interop loop on an STA background thread)
- **Threading:** Dedicated **STA** worker thread for Word Interop; UI updates via `Progress<T>`.

## Key decisions made

### Libraries / Tech
- **Excel:** ClosedXML
- **Word editing:** DocumentFormat.OpenXml (Open XML SDK 3.4.1)
- **PDF generation:** Microsoft Word Interop (`ExportAsFixedFormat`)
- **Constraints:**
    - Word template is a pre-existing mail merge template; merge field format cannot be customized.
    - Merge fields exist only in the **main body** (no headers/footers for now).
    - Missing/blank Excel values become **blank string**.
    - Output: **one PDF per Excel row**.
    - Filename collisions: **overwrite** existing PDFs.

### Data extraction / mapping rules
- Worksheet: default first worksheet; UI allows selecting sheet.
- Header row: default row 1; UI allows choosing row number.
- Values: use **formatted/display values** (`GetFormattedString()`), not raw values.
- Mapping is **case-sensitive**, but supports real-world differences between Excel headers and Word merge field names via normalization:
    - spaces/underscores/hyphens treated as equivalent separators
    - trimming of extra spaces and leading/trailing underscores
    - `%` and `:` removed for matching
    - special Word behavior handled:
        - Word often prefixes fields with `M_` when header starts with `%` or a digit (e.g., `10% PONDERADO` ↔ `M_10_PONDERADO`)
        - Word may join month-year tokens (e.g., `01-2026` ↔ `012026`)

### Execution model
- Per row:
    1) build `record` from Excel (formatted strings)
    2) build output PDF path (`FieldX_FieldY.pdf`, sanitized)
    3) copy template to temp DOCX
    4) merge using OpenXML into temp DOCX
    5) export temp DOCX → PDF using Word Interop
    6) delete temp DOCX
- Continue-on-error: failures are logged per row; run continues.

### UI / UX
- File pickers for template, Excel, output directory.
- Dropdowns for worksheet and FieldX/FieldY naming columns.
- Header row selector (NumericUpDown).
- Log list (`lstLog`) supports multi-select and copy to clipboard (Ctrl+C + context menu).
- “Auto-scan on Run”: Run triggers template scan + validation automatically (manual scan button optional).
